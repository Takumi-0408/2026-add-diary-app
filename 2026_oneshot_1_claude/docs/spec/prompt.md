## 投入プロンプト本文（ここから下をコピペ）

あなたは熟練の Web フロントエンドエンジニアです。同梱の `docs/spec/spec.md` を **唯一の正本仕様**、`docs/plans/2026-05-14-pokens-subagent-implementation.md` を **実装オーケストレーション計画** として、ポケモンになりきり SNS 「ぽけNS」をプロジェクトルートの `index.html` **1 ファイルのみ** で実装してください。

計画ファイルは Phase 1〜4 のタスク分解と並列実行戦略を定義しています。**Task / Agent ツールが利用できる場合は、Phase 1 と Phase 2 を計画どおりサブエージェントで並列ディスパッチ** してください。利用できない場合は Phase 順に逐次実装すれば計画と等価な成果物が得られます。

### 出力要件

- 出力ファイル: プロジェクトルート直下の `index.html` 単一ファイル
- 形式: HTML + 内包 CSS（`<style>`）+ 内包 JS（`<script>`）。外部 JS / CSS / 画像ファイルへの参照は **Google Fonts と PokeAPI のみ** 可
- 動作要件: ブラウザでファイルを直接開く（`file://` プロトコル）だけで完全動作すること
- 禁止事項:
  - JS / CSS フレームワーク（React, Vue, Tailwind 等）の利用
  - ビルドツール（webpack, vite 等）の前提
  - ローカルファイル fetch（PokeAPI 以外のネットワーク不要 fetch）
  - サーバー起動を前提とする実装

### 仕様の参照優先順位

1. `docs/spec/spec.md` の Section 3「MVP スコープ」に **無い** 機能は実装しない
2. Section 4「デザインシステム」のトークン（色 hex / フォントファミリー）は **厳密に再現**
3. Section 5「画面仕様」の各画面要件を 6 画面すべて満たす
4. Section 7「PokeAPI 連携」の通り、初代 151 匹からランダム 5 匹を引く方式で実装
5. Section 9「NPC データ」のテンプレに従って、10 体程度の NPC ポケモン + 15〜25 件の投稿 + 各投稿への 2〜4 コメントをハードコードする

### 実装オーケストレーション

`docs/plans/2026-05-14-pokens-subagent-implementation.md` のフェーズ分けに従う:

- Phase 1（基盤）: Task 1-A〜1-D を **並列**（デザインシステム CSS / 画面 CSS / コア JS / NPC データ）
- Phase 2（画面）: Task 2-A〜2-D を **並列**（オンボーディング / タイムライン+詳細 / 投稿作成 / プロフィール+図鑑）
- Phase 3: オーケストレータが `index.html` スケルトンに全成果物を組み立て
- Phase 4: 仕様適合レビューと機能フロー検査を **並列** に実施し、未解決指摘があれば修正サブエージェントで対応（最大 2 周）

サブエージェントは自分の担当タスク以外の実装本体を読まないこと（インターフェースのみ共有）。

### 仕上がりの優先順位

時間や工数が足りない場合、以下の優先順位で削減してください。**上から順に必ず動かす**:

1. **オンボーディング全 4 ステップが通る**（草むら→5 匹→選択・命名→タイムラインへ）
2. **タイムラインに NPC 投稿が表示される**（PokeAPI スプライトが見える）
3. **投稿作成 → タイムラインへの反映**（localStorage 保存）
4. **投稿詳細 + コメント表示**
5. **プロフィール（自分）画面**
6. **図鑑画面**
7. デザインシステムの細部（注釈、Chrome バー装飾など）
8. 草むら揺れアニメ等の演出
9. Lo-Fi / Mid-Fi 切替（任意機能、なくてよい）

### 実装手順の推奨

1. HTML スケルトン（`<!doctype html>`、フォント読み込み、フレーム要素）
2. CSS 変数（`spec.md` Section 4.1 のカラートークンを `:root` に）
3. 共通コンポーネントのスタイル（フレーム、AppBar、TabBar、Pill、Avatar、Button、PostCard、Segment、GrassTile）
4. 簡易ルーター（hash ベース推奨）と画面切替の枠組み
5. localStorage アクセス薄ヘルパ（partner / posts / seen の読み書き）
6. PokeAPI 取得ヘルパ + キャッシュ
7. NPC ハードコードデータ
8. オンボーディング画面 4 ステップを実装、partner 保存後にタイムラインへ遷移できることを最初に確認
9. タイムライン → 投稿作成 → 投稿詳細を実装
10. プロフィール → 図鑑を実装

### 動作確認の観点

実装完了の判定は `spec.md` Section 13「完成判定（観点）」の 10 項目すべてを満たすこと。

### 自由度

- 細かな余白・行間・ボタン配置の数値は仕様の意図を保つ範囲で調整可
- 文言・コピーは仕様の例を踏襲しつつ、世界観が伝わる範囲で創作可
- 効果音 / 振動 / 物理演出は不要（時間が余ったら任意）

## 別セッションへの引き渡し手順（運用メモ）

実装セッションで以下をセットアップする:

1. このリポジトリを `cd` で開く（または作業 worktree 上）
2. AI に次の 3 つを渡す:
   - 上記「投入プロンプト本文」セクションの内容
   - `docs/spec/spec.md` 全文（正本仕様としてコンテキスト添付）

3. **その他のソースコード（過去デザインの HTML / JSX 等）は添付しない**
4. 生成された `index.html` をブラウザで開いて 13 章「完成判定」を目視確認
